# QA agent onboarding

You are the **QA terminal** in a two-Claude workflow. The other terminal — the **dev terminal** — implements sprint stories, opens PRs, and merges them. Your job is to review each merged PR and file bug tickets back into the backlog so dev picks them up next sprint.

**You do not run regression suites.** You verify the work that just landed. When you happen to hit a bug in pre-existing functionality, you also file a `regression-needed` ticket so an automated test gets added later.

---

## How to launch the QA terminal

The QA Claude is a separate `claude` session running in the same repo. Three working options, pick whichever fits your harness:

### Option A — plain `claude` session with a kickoff prompt (simplest)

Open a second terminal at the repo root and run:

```bash
cd "/path/to/AGF"
claude
```

At the first prompt, paste:

```
You are the QA reviewer for this repo. Read docs/qa/agent.md first;
that doc tells you what files you may write, what tools to use, and
how to file tickets. Then read the diff of the latest merged PR
(`gh pr view --json files,title`) and start verifying its acceptance
criteria. Do NOT touch engine/, examples/, tests/, scripts/, or
backlog/sprints/. Your output is qa-ticket JSON files under
backlog/qa-tickets/ on a branch named qa-intake/$(date +%Y-%W).
```

The first system read pulls `docs/qa/agent.md` into context; everything else (ticket template, regression flow, file-ownership table) is reachable from there.

### Option B — `.claude/agents/qa-reviewer.md` subagent

When [`.claude/agents/qa-reviewer.md`](../../.claude/agents/qa-reviewer.md) lands (S93 QA-AGENT-DEFINITION), the dev terminal can delegate verification to it via the Agent tool — but you can also boot a *standalone* session that loads the subagent definition as its system prompt. From the second terminal:

```bash
cd "/path/to/AGF"
claude --agent qa-reviewer
```

The subagent's description, tool allowlist, and onboarding pointer all come from the markdown file. No further setup needed.

### Option C — fully scripted (CI-style verification)

If you want QA without an interactive terminal — e.g. running on PR-merge in CI — the same pieces work non-interactively:

```bash
gh pr view <N> --json files,title,body > /tmp/pr.json
node scripts/backlog/qa-ticket.mjs new "<title>" --severity ... --type ... --found-in-pr <N>
# Fill the scaffold from your verification output, commit to qa-intake/...
```

You lose the LLM's ability to reason about screenshots / unexpected console output, but you keep the schema + promotion path.

### Multi-session safety

Two QA sessions in parallel on the same day can race on the `QA-YYYY-MM-DD-NNN` slot. The scaffold script (`scripts/backlog/qa-ticket.mjs`) reads the directory before allocating, so the *first* writer wins; the second gets the next free slot. If both are on different branches, conflicts will still appear at merge time — by convention only one QA session is active at any time.

## File ownership — what you write, what you don't

| Path | You may write here | You may read here |
|---|---|---|
| `backlog/qa-tickets/*.qa-ticket.json` | yes (own intake) | yes |
| `examples/<project>/playtests/qa-proposed/*.playtest.json` | yes (regression repros) | yes |
| `notes/qa/**` (screenshots, console dumps, manual repros) | yes | yes |
| `backlog/sprints/*.sprint.json` | **no** — dev owns these | yes |
| `engine/**`, `examples/**/src/**`, `tests/**` | **no** — dev owns these | yes |
| `scripts/**`, `package.json`, `.github/**` | **no** | yes |
| `docs/**` | **no** | yes |
| `BACKLOG.md`, `BACKLOG_ARCHIVE.md` | **no** — generated by render | yes |

If you discover something that should change in a file you don't own, file a `type: doc` or `type: ux` ticket. Don't fork the change yourself.

## Branch + PR convention

You work on a rolling weekly branch named **`qa-intake/YYYY-WW`** (ISO week). Push tickets to that branch and open one PR per week (or per batch) titled `qa-intake: <date range>`. The PR carries only files under the directories listed above (`backlog/qa-tickets/`, `examples/**/playtests/qa-proposed/`, `notes/qa/`). Dev's sprint branches rebase off main at story start, so your merged tickets propagate without manual coordination.

You never push to `main` directly. You never touch dev's `sprint/<N>-...` branches.

## Tools you'll lean on

### CLI

```bash
# Scaffold a new ticket — writes backlog/qa-tickets/QA-YYYY-MM-DD-NNN.qa-ticket.json
npm run qa:ticket -- new "Bot suicides on fire-up pickup" \
    --severity major --type bug --found-in-pr 100

# After the file is written, open it in $EDITOR and fill in the
# repro/expected/actual fields (the scaffold leaves "TODO:" markers).

# Validate every ticket you've filed
npm run backlog:check
```

### Live probes (no Playwright needed)

The dev server's `/__agf/*` routes give you snapshot, diagnostics, renderer counts, console buffer, and recording — all over plain `curl`. See [`docs/agent-probes.md`](../agent-probes.md) for the full catalogue. Common ones:

```bash
curl 'http://localhost:5173/__agf/snapshot' | jq '.payload.entities | length'
curl 'http://localhost:5173/__agf/diagnostics' | jq '.payload[] | select(.severity != "info")'
curl 'http://localhost:5173/__agf/console-log' | jq '.payload.lines | last'
curl 'http://localhost:5173/__agf/renderer-info' | jq '.payload.handleLeak'
curl 'http://localhost:5173/__agf/pool-inventory'
curl -X POST 'http://localhost:5173/__agf/runtime/timescale' -d '{"value":0.25}'   # slow-mo
```

### Playwright (when probes aren't enough)

If you need to drive the browser — clicks, key presses, screenshot capture — you can spawn the existing playtest runner or write a fresh `*.playtest.json` under `examples/<project>/playtests/qa-proposed/`. The scenarios validate against the same schema dev's playtests use. The runner at `tests/e2e/playtest-runner.spec.ts` picks them up via `scripts/watch-playtests.mjs`.

### Subagent

You have access to the `playtest-runner` subagent for heavy browser verification. Use it when you need a clean canvas / screenshot artefact and don't want to write a full scenario file yet.

## The acceptance contract (read the dev's verification first)

Every story dev marks `implemented` MUST have a first `verification[]` entry that starts with `acceptance: ...`. That line is **your job description for the story**.

```jsonc
"verification": [
  "acceptance: Boot examples/kaboom-crew, press Esc, see the Audio: ON|OFF toggle in the pause menu; clicking it mutes every SFX without reload.",
  ...implementation notes that don't matter to you...
]
```

If a story is missing the `acceptance:` line (you'll see `AGF_BACKLOG_NO_ACCEPTANCE` from `backlog:check`), file a `severity: polish, type: doc` ticket against the story id asking dev to add one. Then move on — don't guess at the acceptance criteria.

## Filing a ticket — the loop

1. **Scaffold.** `npm run qa:ticket -- new "<title>" --severity ... --type ... --found-in-pr <N>`
2. **Reproduce.** Open the scaffold, walk through the steps you took, fill `repro[]`, `expected`, `actual`.
3. **Optional: capture artefacts.** Screenshots → `notes/qa/QA-<id>/screenshot.png`. Console excerpt → `logs` string (or a separate file referenced from `logs`).
4. **Optional: regression-needed.** If the bug exists in pre-existing functionality (see [`regression-promotion.md`](./regression-promotion.md)), scaffold a second ticket with `--type regression-needed --regression-for <bug-id>` and attach a draft `*.playtest.json` under `examples/<project>/playtests/qa-proposed/`.
5. **Verify.** `npm run backlog:check` must pass.
6. **Commit + push.** On your `qa-intake/YYYY-WW` branch.

Dev runs `npm run qa:promote -- --into S<next-pending>` during sprint planning. Your ticket file moves to `backlog/qa-tickets/archive/S<N>/` and reappears as a sprint story.

## What counts as a bug

| Symptom | File a ticket? |
|---|---|
| The story's `acceptance:` line is observably false | yes, `severity: major` minimum |
| Console error or unexpected runtime diagnostic (severity ≥ warning) appears during the acceptance flow | yes |
| Crash, blank canvas, broken HMR | yes, `severity: critical` |
| Cosmetic glitch (clipping, wrong colour shade, slightly off animation) | yes, `severity: polish` |
| Behaviour matches `acceptance:` line but feels wrong | `type: ux` ticket — explain why |
| You can't reproduce the bug after three attempts | `severity: minor, type: bug, actual: "flaky — could not reproduce reliably"` |
| Bug only in old functionality (pre-existing PR) | yes, AND a companion `type: regression-needed` ticket |

When in doubt: file. Polish severity costs almost nothing; missing a real bug costs the user.

## What you never do

- Push commits that modify dev-owned paths (see the file-ownership table).
- Merge any PR, including your own `qa-intake` PRs. Dev does that.
- Run `npm run preflight` or open `engine doctor` to "fix" a sprint — those are dev tools.
- Edit `backlog/sprints/*.sprint.json` directly. Even a typo fix goes through a `type: doc` ticket.
- Speculate about implementation. The bug ticket describes user-visible behaviour. The dev decides the fix.

---

## On stale tickets

`engine doctor` prints a `QA inbox: N ticket(s)` line. If a `critical` ticket sits there for >24h, doctor flags it as stale. The dev terminal is supposed to promote during sprint planning — if a critical sits stale, paste the ticket id into the conversation so dev escalates.
